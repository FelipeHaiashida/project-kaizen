## Codebase Patterns
(Lições reutilizáveis descobertas ao longo das iterações. Leia antes de começar.)

- **Stack fixada:** Next.js 14.2 (App Router) + React 18 + TypeScript. NÃO usar Next 15 / React 19.
- **Tailwind é v3.4** (não v4). shadcn/ui está configurado no estilo CLÁSSICO:
  - `app/globals.css` usa variáveis HSL (`--primary: 262 83% 58%` etc.) com `@tailwind base/components/utilities`. NÃO usar oklch nem `@import "tailwindcss"`.
  - `tailwind.config.ts` mapeia as cores via `hsl(var(--...))`, `darkMode: ["class"]`, plugin `tailwindcss-animate`.
  - Componentes shadcn usam `@radix-ui/*` + `class-variance-authority` + `cn()` de `lib/utils.ts`. NÃO usar `@base-ui/react`.
  - Cor primária da marca = violeta ClickUp `#7C3AED` = `hsl(262 83% 58%)`.
  - Ao rodar `npx shadcn@latest add <comp>`: a CLI mais nova pode gerar código estilo "base-nova"/Tailwind v4. Se isso acontecer, adaptar para o estilo clássico (Radix + HSL) antes de commitar.
- **Prisma é v6** (fixado). Prisma 7 removeu `url`/`directUrl` do schema (exige driver adapters + prisma.config.ts) — por isso ficamos no 6. Generator: `prisma-client-js`. Cliente importado de `@prisma/client`.
- **Variáveis de ambiente:**
  - `.env` (gitignored) → `DATABASE_URL` e `DIRECT_URL`. **O Prisma CLI só lê `.env`, não `.env.local`.**
  - `.env.local` (gitignored) → segredos de app (NextAuth, Supabase, Resend). Lido pelo Next.js.
  - `.env.example` (versionado) → documenta todas as variáveis.
- **Banco JÁ está no ar (Supabase).** Projeto `kaizen` ref `kyrvyqtjlrxgzbrhxjiz` (ca-central-1),
  16 tabelas criadas. `DATABASE_URL` = pooler transaction (6543, `?pgbouncer=true`), `DIRECT_URL` = pooler session (5432).
  - **Usar `npx prisma db push` para sincronizar o schema** (NÃO `migrate dev` — o shadow database falha no pooler do Supabase). Sem histórico de migrations por enquanto; ok para esta fase.
  - Testar conexão avulsa: `node --env-file=.env -e "..."` (não há `dotenv` instalado; o Node 24 tem `--env-file`).
  - Falta `SUPABASE_SERVICE_ROLE_KEY` no `.env.local` (necessária só a partir da S03/Storage).
- **Integração MCP do Supabase aponta para OUTRA conta** (anabeatrizleite235), não a do projeto atual. Portanto NÃO dá para usar as tools MCP (list_tables, advisors, logs, apply_migration) contra este banco — usar Prisma diretamente.
- **Autenticação (NextAuth v5 / Auth.js beta):** já configurada. Padrão em uso:
  - `lib/auth.config.ts` = config edge-safe (SEM Prisma/bcrypt) usada pelo `middleware.ts`. `lib/auth.ts` = config completa com o provider `Credentials` (usa Prisma + bcryptjs). Exporta `{ handlers, auth, signIn, signOut }`.
  - Sessão **JWT** (`strategy: "jwt"`). `secret` vem de `NEXTAUTH_SECRET`; `trustHost: true`.
  - Proteção de rotas: callback `authorized` em `auth.config.ts` (públicas: `/`, `/login`, `/register`, `/invite/*`; resto exige login). `middleware.ts` = `NextAuth(authConfig).auth` com matcher que exclui `api/auth` e assets.
  - `id` do usuário propagado via callbacks `jwt`/`session` + augmentação de tipos em `types/next-auth.d.ts`.
  - Para pegar a sessão em Server Component: `const session = await auth()`. Logout: `<form action={async()=>{"use server"; await signOut({redirectTo:"/login"})}}>`.
  - **Senhas com `bcryptjs`** (puro JS — evita build nativo no Windows), 10 rounds.
  - Mutations de auth em `app/(auth)/actions.ts` (Server Actions) — validam com Zod e retornam `{ error }`; o redirect do `signIn` é repassado (re-throw se não for `AuthError`).
- **Zod é v4** — `z.string().email()` ainda funciona; erros via `parsed.error.issues[0].message`.
- **Componentes shadcn adicionados à mão** (estilo clássico Radix v3): `input`, `label` (usa `@radix-ui/react-label`), `card`. Continuar escrevendo à mão em vez de `npx shadcn add`.
- **Grupos de rota:** `app/(auth)/` (login/registro, layout centralizado) e `app/(app)/` (área logada, layout com header + logout, valida `auth()`). Route groups não afetam a URL.
- **Verificação no preview:** `preview_screenshot` dá timeout neste ambiente, mas **`preview_snapshot` (árvore de acessibilidade) funciona** e é ótimo para verificar texto/estrutura. Fluxos podem ser testados com `preview_fill`/`preview_click` + checagem no banco via `node --env-file=.env`.
  - **Cuidado com seletores de submit:** o header da área logada tem um `<form>` de logout com `button[type=submit]`. Um seletor genérico `form button[type=submit]` clica no "Sair" (primeiro do DOM). Use seletores escopados, ex.: `main form button[type=submit]` ou `form:has(#campo) button[type=submit]`.
  - Sessão JWT persiste no cookie mesmo após deletar o usuário do banco — ao re-testar, faça **logout** (botão Sair) antes de registrar outro usuário.
- **Supabase Storage:** cliente admin (service_role) em `lib/supabase/admin.ts` (tem `import "server-only"` — nunca importar no client). Bucket **`avatars`** (público) já criado. Upload em Server Action recebendo `FormData` (File → `Buffer.from(await file.arrayBuffer())`, `upsert:true`); URL pública via `getPublicUrl` + cache-buster `?v=Date.now()`.
- **Toasts:** `sonner`. `<Toaster richColors />` já está no root `app/layout.tsx`. Usar `toast.success/error` nos client components após Server Actions.
- **JWT vs dados frescos:** a sessão JWT não reflete mudanças de nome/foto até relogar. Onde precisar de dados atuais (ex.: header em `app/(app)/layout.tsx`), buscar do banco por `session.user.id`. Após mutations, chamar `revalidatePath("/", "layout")` + `router.refresh()`.
- **Upload por FormData + react-hook-form:** o RHF valida os campos de texto; no `onSubmit` monta-se um `FormData` manualmente (campos + File do input escondido) e chama-se a Server Action. Input `type=file` é escondido e acionado por um botão via `ref`.
- **Multi-tenant / Workspaces:** o "workspace ativo" é guardado no cookie **`kaizen.ws`** (= slug). Helpers em `lib/workspace.ts`: `getActiveWorkspace()` (server: lê cookie, fallback pro 1º; retorna `{workspace, role, memberships}`), `getUserMemberships()`. Setar o cookie só em Server Action (`cookies().set`). Papéis: helper PURO em `lib/roles.ts` (`canManageWorkspaceRole`, `ROLE_LABELS`) — importável no client (o `lib/workspace.ts` é server-only).
  - **Shell da área logada** (`app/(app)/layout.tsx`): sidebar com `WorkspaceSwitcher` + nav; **redireciona pra `/onboarding` se o usuário não tem workspace** (é assim que o onboarding "no primeiro login" é imposto). `/onboarding` fica em grupo próprio `app/(onboarding)/` (fora do shell) e sempre mostra o form de criar (serve também pra criar workspaces adicionais).
  - Server Actions de workspace em `lib/actions/workspace.ts`: create/update/switch/removeMember/getInviteToken.
  - **Storage genérico:** `lib/storage.ts` `uploadImage(bucket, path, file)` (reusa o admin client). Bucket `workspace-logos` (público) para logos.
  - **Link de convite (S04):** `getInviteToken()` cria/reaproveita uma `Invitation` **aberta** (email=null) válida 7 dias e o client monta `${origin}/invite/${token}`. O model `Invitation.email` virou opcional pra suportar link aberto (S04) e convite por email (S05). A página `/invite/[token]` de aceite é da S05.
- **Radix dropdown não abre com `.click()` sintético** no preview — usar `preview_click` (evento real) no trigger; depois os `[role=menuitem]` podem ser lidos/clicados via `preview_eval`. Para `window.confirm` em testes, sobrescrever `window.confirm = () => true` antes de clicar. Navegação no preview: `preview_eval` com `window.location.href=...` (não há tool de navigate). **Radix Dialog trigger** (asChild Button), por outro lado, **abre com `.click()` sintético** via `preview_eval`.
- **Email (Resend):** `lib/email.ts` — client só instanciado se `RESEND_API_KEY` existe; `sendInvitationEmail` **nunca lança** (retorna `{sent, error}`) pra não quebrar o fluxo se o email falhar. Remetente via `EMAIL_FROM` (padrão `onboarding@resend.dev`). **Modo teste da Resend (sem domínio verificado) só entrega pro email dono da conta** — envio pra terceiros dá 403; por isso as Server Actions criam o convite mesmo se o email falhar e devolvem o link pra copiar. Para produção: verificar domínio em resend.com/domains e trocar `EMAIL_FROM`.
- **Convites:** `Invitation.email` = null → link aberto (S04, reusável, não consome); email preenchido → convite direcionado de **uso único** (marca `acceptedAt` ao aceitar). Actions em `lib/actions/invitation.ts`: `inviteMember`, `revokeInvitation`, `acceptInvitation`. Página pública `/invite/[token]` valida (inexistente/aceito/expirado) e, deslogado, manda pra `/login|/register?callbackUrl=/invite/token`.
- **`redirectTo`/`callbackUrl` no auth:** `authenticate`/`registerUser` aceitam `redirectTo` opcional (sanitizado por `safeRedirect` — só caminhos internos). Páginas `login`/`register` leem `searchParams.callbackUrl`. É assim que o fluxo "aceitar convite como novo usuário" volta pro convite após registrar/entrar.
- **Projetos:** rota base `app/(app)/[workspace]/[project]/page.tsx` (`[workspace]` = slug, `[project]` = id). Segmentos estáticos (dashboard/settings) têm precedência sobre o dinâmico `[workspace]`, sem conflito. Ao criar projeto, `createProject` já semeia **4 status padrão** (A Fazer/Em Progresso/Em Revisão/Concluído) + **lista padrão "Tarefas"** (`lib/project.ts` `DEFAULT_STATUSES`/`DEFAULT_LIST_NAME`) — base pras S07/S08.
  - `deleteProject` faz **cascata manual** numa `db.$transaction` (assignees→tags→attachments→reactions→comments→notifications→tasks→lists→statuses→tags→customFields→project). Não há `onDelete: Cascade` no schema; deletar em cadeia é responsabilidade da action.
  - Sidebar de projetos = client (`ProjectsNav` + `ProjectNavItem`): busca, seção Arquivados colapsável, menu de contexto (editar/arquivar/excluir). `ProjectFormDialog` (criar/editar) com seletor de cor/emoji e visibilidade. `visibility` (PUBLIC/PRIVATE) é armazenada e mostra 🔒; enforcement fino de acesso fica para depois.
  - `getWorkspaceProjects`/`getProjectForWorkspace` em `lib/project.ts`. Ícones/cores em `lib/validations/project.ts`.
- **Listas:** actions em `lib/actions/list.ts` (create/update/delete/reorder), validação em `lib/validations/list.ts`. `deleteList` faz cascata manual das tarefas. `reorderLists(projectId, orderedIds)` grava `order` por índice via `updateMany` (where id+projectId).
  - UI: `ListsView` (client, `@dnd-kit` — `DndContext`+`SortableContext` vertical) renderiza `SortableListSection` por lista (grip pra arrastar, header cor+nome+contagem, menu renomear/excluir). `ListFormDialog` cria/edita (nome+cor). Estado local sincronizado com props via `useEffect`.
  - **`SortableListSection` aceita `children`** (área de tarefas) — a S08 injeta as tarefas por lista via prop `bodies` do `ListsView`. Sem children mostra "Nenhuma tarefa".
- **dnd-kit instalado** (`@dnd-kit/core|sortable|utilities`). Reordenar via drag é implementado; **não dá pra simular o arraste no preview headless de forma confiável** — validar reorder manualmente ou por teste de action.
- **NÃO rodar `npm run build` com o dev server (preview) rodando** — os artefatos de produção colidem com o cache dev e quebram o dev server ("Cannot find module './XXX.js'"). Se acontecer: parar o preview, `rm -rf .next`, reiniciar. Rodar o build só com o preview parado.
- **Tarefas:** actions em `lib/actions/task.ts` (createTask com parentId opcional pra subtarefa, updateTask com responsáveis, setTaskStatus, deleteTask com cascata de subtarefas). Prioridade meta em `lib/tasks.ts` (`PRIORITIES`/`PRIORITY_MAP`). Ao criar tarefa usa o **primeiro status** do projeto; "concluído" = **último status** (maior order).
  - UI: `ListsView` passa `bodies[listId] = <TasksList/>` (Server Component com filhos client) — **Server Components podem ser passados como prop (não só children) para Client Components**. `TasksList` = TaskRow (client, abre `TaskDetailSheet`) + `InlineTaskCreate` (Enter cria).
  - `TaskDetailSheet` (painel lateral via `components/ui/sheet.tsx`, Radix Dialog ancorado à direita): título, descrição (**TipTap** via `components/ui/rich-text-editor.tsx`, `immediatelyRender:false`), status/prioridade (select nativo), responsáveis (chips toggle com avatar), vencimento (`input type=date`), subtarefas (checkbox conclui via status, add por Enter, excluir), excluir tarefa. Campos principais em `useState` local (persistem entre `router.refresh()`); subtarefas vêm do prop `task` (atualizam no refresh).
  - Tipos compartilhados em `components/task/types.ts`.
- **TipTap** instalado (`@tiptap/react|starter-kit|pm`, v3). Editor client-only com `immediatelyRender:false` (evita erro de SSR no Next). Descrição salva como HTML em `Task.description`. Sem plugin de typography — estilos de lista/código aplicados via classes utilitárias no `editorProps`.
- **Simular teclado/select no preview:** inputs controlados → `preview_fill`; `<select>` controlado e Enter → `preview_eval` (setar `.value` com o native setter + `dispatchEvent(new Event('change'))`, ou `dispatchEvent(new KeyboardEvent('keydown',{key:'Enter'}))`). `contenteditable` do TipTap não é preenchível por `preview_fill`. **ATENÇÃO:** o truque de native-setter+change em `<select>` controlado do React às vezes NÃO propaga pro estado — preferir `preview_fill` (que dispara o onChange do React de forma confiável). Dar `id`s aos controles ajuda a mirar com `preview_fill`.
- **Tags e Campos customizados (S09):** Tag/CustomField já existiam no schema; adicionei `CustomFieldValue` (valor por tarefa, `@@unique([fieldId,taskId])`), `CustomField.order` e `Task.estimateHours`. Actions em `lib/actions/tag.ts` e `lib/actions/custom-field.ts` (`setTaskTags`, `setTaskFieldValue`, `setTaskEstimate`). Gerenciadores em `/[workspace]/[project]/settings` (`TagManager`, `CustomFieldManager`). No `TaskDetailSheet`: toggles de tag, inputs por tipo de campo (`field-<id>`), estimativa. Valores de campo salvos como **string** (checkbox = "true"/""). Filtro por campos fica pra S11.
- **Anexos (S10):** bucket **`task-attachments`** (público, 10MB). Actions em `lib/actions/attachment.ts` (`uploadAttachment` recebe FormData/File; `deleteAttachment` remove do Storage E do banco — extrai o path da URL pública pelo marcador `/object/public/<bucket>/`). Componente `TaskAttachments` no `TaskDetailSheet`: dropzone (drag-drop) + botão, preview inline de imagens (`<img>`), lista com nome/tamanho/data, download (`<a download>`), excluir. Seletor de arquivo do navegador não é acionável no preview — validar upload por script.
- **Visualização em Lista (S11):** `components/task/task-list-view.tsx` é a view PRINCIPAL da página do projeto (substituiu a antiga `ListsView`/`TasksList`/`TaskRow`, que foram DELETADAS — código morto). Recebe tarefas achatadas (`TaskViewItem` = TaskListItem + listId/listName). Recursos: toggle agrupar por Lista/Status, busca por título, filtros (responsável/prioridade/status/vencimento), ordenação (título/prioridade/vencimento/status + direção), colunas configuráveis (dropdown), criação inline por grupo (em grupo de status, cria na 1ª lista e seta o status), checkbox de concluir (status = último), seleção múltipla + barra de ações em lote. Detalhe da tarefa: UM único `TaskDetailSheet` controlado por `openTaskId` (não um por linha). Reorder de listas (dnd) preservado no `ListGroupCard` quando agrupado por lista.
  - Bulk actions em `lib/actions/task-bulk.ts` (`bulkSetStatus`/`bulkMove`/`bulkDelete`), com filtro de segurança por workspace. Selects de bulk são **não-controlados** (defaultValue) — aí o dispatch sintético de `change` funciona no preview.
  - **`Set` não pode ser iterado com spread** no target atual do TS — usar `Array.from(set)` (não `[...set]`).
- **Singleton do Prisma:** importar `{ db }` de `lib/db.ts` (NÃO instanciar `new PrismaClient()` avulso).
- **Tipos globais:** importar de `@/types` (reexporta models/enums do Prisma).
- **Convenções:** arquivos em kebab-case, componentes em PascalCase. Rotas autenticadas em `/(app)/`, públicas `/login` `/register` `/invite/[token]`. Mutations via Server Actions. Validação com Zod.
- **npm tem allow-scripts ativo:** postinstall de pacotes (ex.: engines do Prisma) pode ser bloqueado com aviso. `npx prisma generate` funciona mesmo assim; se um engine faltar, rode `npm approve-scripts`.
- **Checks de qualidade:** `npm run typecheck`, `npm run lint`, `npm run build`, `npm run format`. Todos devem passar antes do commit.
- **Preview/screenshot:** o `preview_screenshot` deu timeout neste ambiente mesmo com o server retornando 200. Verificação via `preview_logs` (HTTP 200) + `npm run build` é suficiente.
